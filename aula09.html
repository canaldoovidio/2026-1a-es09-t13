BLOCO 1

Padrões de Integração Empresarial

Enterprise Integration Patterns (EIP) + Tecnologias do artigo

EIP

5 Padrões Essenciais de Integração

swap_horiz

Message Channel

Canal dedicado entre sistemas. Cada tipo de dado (pedido, NF-e, pagamento) tem seu próprio canal.

alt_route

Message Router

Decide para onde enviar a mensagem com base em regras. Ex.: NF-e de SP vai para SEFAZ-SP, de RJ para SEFAZ-RJ.

transform

Message Translator

Converte formatos entre sistemas. Ex.: JSON do app → XML da SEFAZ, ou vice-versa.

schema

Canonical Data Model

Modelo de dados único e compartilhado. Cada sistema traduz para/de o modelo canônico em vez de N × N traduções.

bolt

Event-Driven Consumer

Sistemas reagem a eventos assíncronos. Ex.: “pedido criado” dispara atualização de estoque, NF-e e notificação.

Fonte: Hohpe & Woolf, Enterprise Integration Patterns (2003) — enterpriseintegrationpatterns.com

ARTIGO — Seção 3

Tecnologias de Integração (ESI)

api

APIs (REST, GraphQL)

Pedra angular da integração moderna. OpenAPI, lifecycle management, API-first design. “APIs como produtos.”

mediation

Middleware / iPaaS

De ESBs monolíticos a plataformas cloud-native. Transformação, roteamento, conversão de protocolos.

bolt

Event-Driven (EDA)

Pub/sub, event sourcing, Kafka. Acoplamento fraco, escalabilidade, processamento em tempo real.

cloud

Cloud-Native

Containers, serverless, event mesh. Integração elástica com resiliência por redundância.

format_quote “The API-first approach to system design has gained traction as it prioritizes interoperability from inception rather than treating integration as an afterthought.” — Adusumilli, 2025

CASO 1

SEFAZ — Retry + Circuit Breaker

error_outline Problema

SEFAZ tem janelas de manutenção não anunciadas. O sistema precisa continuar operando mesmo sem emissão de NF-e.

check_circle Solução

CB com timeout 3s + retry exponencial + fallback para fila de contingência.

@circuit_breaker(failure_threshold=3, reset_timeout=30)
def emitir_nfe(dados):
    try:
        response = sefaz_client.post("/nfe", json=dados, timeout=3)
        return response.json()
    except Timeout:
        fila_contingencia.enqueue(dados)
        return {"status": "contingencia", "msg": "NF-e em fila"}

Padrões EIP: Message Channel (fila de contingência) + Message Router (roteamento por estado UF)

CASO 1

    And o circuit breaker está CLOSED
    When 3 tentativas de emissão falham
    Then o CB muda para OPEN
    And NF-e é enviada para fila de contingência

Step Definitions

@given('a SEFAZ está em manutenção')
def step_sefaz_manutencao(context):
    context.sefaz = MagicMock()
    context.sefaz.post.side_effect = TimeoutError(
        "SEFAZ indisponível - janela de manutenção"
    )

@then('NF-e é enviada para fila de contingência')
def step_fila_contingencia(context):
    context.fila.enqueue.assert_called_once()
CASO 2

BACEN/PIX — Timeout + Bulkhead

error_outline Problema

PIX deve confirmar em < 10s por regulação do BACEN. Falhar não é opção.

check_circle Solução

Timeout estrito de 8s + bulkhead isolando thread pool de pagamentos.

# Bulkhead: pool isolado para PIX
pix_executor = ThreadPoolExecutor(max_workers=10, thread_name_prefix="pix")

async def confirmar_pix(transacao_id):
    future = pix_executor.submit(bacen_client.confirmar, transacao_id)
    try:
        return future.result(timeout=8)  # Timeout regulatório
    except TimeoutError:
        return {"status": "timeout", "acao": "reverter"}

Padrões EIP: Message Channel (pool isolado) + Event-Driven Consumer (reversão automática por evento)

ANÁLISE

Análise Comparativa das Integrações

Integração Tática Padrão EIP Trade-off Métrica
SEFAZ Retry + CB Channel + Router Complexidade vs disponibilidade NF-e emitidas/hora
BACEN Timeout + Bulkhead Channel + Event-Driven Recursos fixos vs isolamento p95 < 8s
Transportadora Polling + Cache Translator + Channel Dados stale vs rate limit Cache hit ratio
ERP Sync + Retry Canonical Model Latência vs consistência Sync success rate

Cada integração tem perfil diferente de falha. A tática depende do SLA do serviço externo.

PROMPTS

Construindo Prompts com Pensamento Crítico

O Copilot gera rascunhos, não arquitetura final. O valor está no que vocês pedem e no que refinam depois.

1

Contextualize o domínio

“Somos um ERP que integra com SEFAZ, BACEN e transportadoras. O sistema precisa operar 24/7.”

2

Especifique restrições e RNFs

“PIX deve confirmar em <10s. SEFAZ tem janelas de manutenção. Rate limit de 100 req/min na transportadora.”

3

Nomeie os padrões que quer ver

“Use Message Router para rotear por UF, Message Translator para JSON↔XML, Event Bus para eventos de domínio.”

4

Defina o papel da IA explicitamente

“Inclua componente de IA que analisa logs do Event Bus e alerta sobre anomalias de latência.”

5

Peça o tipo de diagrama certo

“Gere como C4 Context para visão geral, OU Sequence para fluxo de criação de pedido com fallback.”

warning Armadilha: aceitar o diagrama do Copilot sem questionar. Sempre pergunte: “Este diagrama reflete o meu projeto ou é genérico?” Refine até que ele represente as integrações reais do ASIS.
