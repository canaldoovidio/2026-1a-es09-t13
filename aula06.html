PARTE 1

Engenharia de Integração

Conceitos de integrabilidade — SEI/CMU (Kazman et al., 2020)

CONCEITO

O que é Integração de Software?

Integrabilidade é a capacidade de uma arquitetura de antecipar e facilitar a integração de componentes separados em um sistema funcional. — SEI/CMU, Kazman et al., 2020

Size (Tamanho)

Quantidade de dependências potenciais entre componentes {Ci} e o sistema S. Quanto mais dependências, maior o custo de integração.

Distance (Distância)

Dificuldade de resolver diferenças em cada dependência. Inclui dimensões: sintática, semântica de dados, semântica comportamental, temporal e de recursos.

Complexidade = ∑ f(size × distance) — O objetivo das táticas de integrabilidade é reduzir size e distance.

INTEGRAÇÃO

Integração Interna vs Externa

account_tree

Integração Interna

Módulos do próprio sistema que se conectam entre si: services, repositories, controllers.

Exemplo repo: DeviationAnalyzer → RouteOptimizer → DistanceCalculator

Exemplo ASIS: NotaFiscalService → CalculoImpostoService → ValidadorCNPJ

cloud

Integração Externa

Comunicação com serviços de terceiros: APIs, bancos de dados, provedores governamentais.

Exemplo repo: Flask app → serviço externo (simulado com retry/circuit breaker)

Exemplo ASIS: App → SEFAZ, App → BACEN, App → Gateway de Pagamento

Fluxo PIX (exemplo): Smartphone → App → API interna → Conta Corrente → BACEN. Cada seta = um ponto de integração que precisa ser aferido.

SEI/CMU

Dimensões de Distância na Integração

code

Sintática

Diferenças em tipos de dados, formatos, protocolos. Ex: JSON vs XML, REST vs SOAP, int vs float.

data_object

Semântica de Dados

Mesmos dados interpretados de forma diferente. Ex: distância em metros vs quilômetros, datas em formatos diferentes.

psychology

Semântica Comportamental

Comportamentos diferentes conforme estado do sistema. Ex: componente A espera push, componente B faz pull.

schedule

Temporal

Diferenças em taxas de operação e expectativas de timing. Ex: timeout de 50ms vs resposta em 200ms.

memory

Recursos

Suposições sobre recursos compartilhados: memória, CPU, bandwidth, conexões de banco.

TÁTICAS

Táticas de Integrabilidade (SEI/CMU)

Categoria Tática O que faz
Limitar Dependências Encapsulate Interface explícita que esconde detalhes internos do módulo
Use Intermediary Quebra dependência direta entre módulos (bus, proxy, adapter)
Restrict Comm. Paths Restringe quais módulos podem se comunicar entre si
Abstract Common Services Esconde módulos similares atrás de uma abstração comum
Adhere to Standards Usa padrões reconhecidos (REST, JSON Schema, OAuth)
Adaptar Tailor Interface Adiciona/remove capacidades de uma interface sem mudar implementação
Configure Behavior Componente configurável para interagir com diferentes parceiros
Coordenar Orchestrate Coordena invocação de serviços para que não se conheçam
Manage Resources Governa acesso a recursos computacionais compartilhados
PATTERNS

Patterns de Integração

SOA (Service-Oriented Architecture)

Serviços loosely coupled com interfaces padronizadas. Usa: Encapsulate, Adhere to Standards, Orchestrate, Manage Resources.

Tradeoff: Complexidade e latência adicionais.

Broker

Componente intermediário que coordena comunicação entre serviços distribuídos. Usa: Encapsulate, Use Intermediary, Restrict Comm. Paths.

Tradeoff: Latência por indireção.

Publish-Subscribe

Comunicação assíncrona via eventos. Publishers e subscribers não se conhecem. Usa: Encapsulate, Use Intermediary, Abstract Common Services.

Tradeoff: Testabilidade mais difícil, segurança.

Adapter

Intermediário que traduz interface de um componente para outra. Resolve diferenças sintáticas e semânticas. Usa: Encapsulate, Use Intermediary, Tailor Interface.

Tradeoff: Latência pela transformação.

INTEGRAÇÃO INTERNA

Services — Módulos que se Conectam

Python src/services/deviation_analyzer.py (código real do repo) 
class DeviationAnalyzer:
    """Analisa se há desvio de rota > 6 km por 3 atualizações consecutivas."""

    def __init__(self, route_optimizer=None):
        self.consecutive_deviations = 0
        self.deviation_detected = False
        self.notification_sent = False
        self.route_optimizer = route_optimizer  # Injeção de dependência!

    def process_location_update(self, truck_id, distance_from_route):
        if distance_from_route > 6.0:
            self.consecutive_deviations += 1
        else:
            self.consecutive_deviations = 0

        if self.consecutive_deviations >= 3 and not self.deviation_detected:
            self.deviation_detected = True
            self._send_notification(truck_id)
            self._request_alternative_route(truck_id)  # Integração interna!

    def _request_alternative_route(self, truck_id):
        if self.route_optimizer:
            self.route_optimizer.request_alternative(truck_id)  # Chama RouteOptimizer
lightbulb

Tática Encapsulate: Cada service tem responsabilidade única. Injeção de dependência permite substituir route_optimizer por um mock nos testes.

Funcionalidade: Detecção de Desvio de Rota

Python (Behave) features/steps/detect_deviation_steps.py 
from behave import given, when, then
from src.services.deviation_analyzer \
    import DeviationAnalyzer
from src.services.route_optimizer \
    import RouteOptimizer

@given('que o caminhão "{truck_id}" está em rota')
def step_truck_in_route(context, truck_id):
    context.truck_id = truck_id
    context.route_optimizer = RouteOptimizer()
    context.deviation_analyzer = \
        DeviationAnalyzer(
            route_optimizer=context.route_optimizer)

@when('recebo 3 updates com desvio de 7 km')
def step_receive_deviations(context):
    for d in [7.0, 7.0, 7.0]:
        context.deviation_analyzer \
            .process_location_update(
                context.truck_id, d)

@then('devo registrar o desvio')
def step_assert_detected(context):
    assert context.deviation_analyzer \
        .deviation_detected is True

Testes Unitários — pytest + MagicMock

Python tests/unit/test_deviation_analyzer.py (código real do repo) 
import pytest
from unittest.mock import MagicMock
from src.services.deviation_analyzer import DeviationAnalyzer

@pytest.fixture
def deviation_analyzer():
    return DeviationAnalyzer()

def test_no_deviation_detected(deviation_analyzer):
    # Com distâncias abaixo de 6 km, não deve disparar alerta
    for d in [3.0, 4.5, 5.9]:
        deviation_analyzer.process_location_update("CX123", d)
    assert deviation_analyzer.deviation_detected is False

def test_deviation_detected_after_three_updates(deviation_analyzer):
    # Mock de notificação e rota — tática Encapsulate + Use Intermediary
    deviation_analyzer._send_notification = MagicMock()
    deviation_analyzer._request_alternative_route = MagicMock()

    for d in [7.0, 7.1, 7.0]:
        deviation_analyzer.process_location_update("CX123", d)

    assert deviation_analyzer.deviation_detected is True
    deviation_analyzer._send_notification.assert_called_once()
    deviation_analyzer._request_alternative_route.assert_called_once()

INTEGRAÇÃO EXTERNA

Retry + Circuit Breaker — Resiliência

Python src/services/retry_circuit_breaker.py (código real do repo) 
# Estados possíveis do Circuit Breaker
CLOSED = "closed"      # Normal — chamadas passam
OPEN = "open"          # Aberto — rejeita chamadas (protege serviço)
HALF_OPEN = "half-open" # Teste — permite 1 chamada para verificar recuperação

circuit_state = CLOSED
open_duration = 3  # segundos antes de tentar HALF_OPEN

async def circuit_breaker_function():
    global circuit_state, time_opened, test_in_half_open
    current_time = time.time()

    if circuit_state == OPEN:
        if (current_time - time_opened) >= open_duration:
            circuit_state = HALF_OPEN      # Tenta recuperar
        else:
            raise Exception("Circuit is OPEN")

    try:
        result = await delayed_function()    # Chama serviço externo
    except Exception as e:
        circuit_state = OPEN               # Falhou → abre circuito
        time_opened = time.time()
        raise e

    if circuit_state == HALF_OPEN:
        circuit_state = CLOSED             # Recuperou → fecha circuito
    return {"message": result, "circuit_state": circuit_state}

Tática Use Intermediary: O circuit breaker é um intermediário entre a aplicação e o serviço externo. Protege contra sobrecarga quando o serviço está com problemas.

Python tests/carga/locustfile.py (código real do repo) 
from locust import HttpUser, task, between

class MyUser(HttpUser):
    wait_time = between(1, 2)  # Espera 1-2s entre tasks

    @task(2)  # Peso 2 — executado 2x mais que o outro
    def test_retry_example(self):
        with self.client.get("/retry-example", catch_response=True) as response:
            if response.status_code != 200:
                response.failure(f"Unexpected: {response.status_code}")
            else:
                response.success()

    @task(1)
    def test_circuit_breaker_example(self):
        with self.client.get("/circuit-breaker-example", catch_response=True) as response:
            if response.status_code != 200:
                response.failure(f"Unexpected: {response.status_code}")
            else:
                response.success()

@integracao_interna

@integracao_interna @falha
Cenário: Funciona sem RouteOptimizer (degradação graciosa)

from unittest.mock import MagicMock

@given('que o DeviationAnalyzer está conectado ao RouteOptimizer')

    context.analyzer = DeviationAnalyzer(route_optimizer=context.mock_optimizer)

@then('o RouteOptimizer.request_alternative deve ser chamado')
def step_assert_integration(context):
    context.mock_optimizer.request_alternative.assert_called_once_with("CX123")
EXEMPLO 2

Aferindo Integração Externa (Retry)

Gherkin + Python Retry — serviço externo com falhas temporárias 
@integracao_externa @retry
Cenário: Serviço responde após falhas temporárias
  Dado que o serviço externo falha nas 2 primeiras chamadas
  E responde com sucesso na 3ª chamada
  Quando o sistema chama o endpoint /retry-example 3 vezes
  Então as 2 primeiras respostas têm status 500
  E a 3ª resposta tem status 200

@integracao_externa @retry
Cenário: Delay dobra a cada chamada (exponential backoff)
  Dado que o delay inicial do serviço é 20ms
  Quando faço 3 chamadas consecutivas ao /retry-example
  Então o tempo de cada resposta é aproximadamente o dobro do anterior

# ── Step Definition ──
@given('que o serviço externo falha nas 2 primeiras chamadas')
def step_setup_failures(context):
    context.mock_service = MagicMock()
    context.mock_service.side_effect = [
        Exception("Service failing"),
        Exception("Service failing"),
        "Service is responding in 20 ms"
    ]

@then('a 3ª resposta tem status 200')
def step_assert_recovery(context):
    assert context.responses[2].status_code == 200

    cb.delay = cb.baseline  # Reset delay to 20ms

@then('a resposta indica estado "open"')
def step_assert_open(context):

  @seguranca @integracao @resistir

RASTREABILIDADE

Aferindo Rastreabilidade de Integração

Gherkin + Python Correlation ID e logs estruturados entre serviços 
@rastreabilidade @integracao
Cenário: Correlation ID propagado entre serviços
  Dado que uma requisição chega com header X-Correlation-ID "abc-123"
  Quando o DeviationAnalyzer processa a requisição
  E chama o RouteOptimizer
  Então o RouteOptimizer recebe o mesmo X-Correlation-ID "abc-123"
  E todos os logs contêm "abc-123"

@rastreabilidade @integracao
Cenário: Logs estruturados em cada ponto de integração
  Dado que o sistema processa uma detecção de desvio
  Quando a integração DeviationAnalyzer → RouteOptimizer é executada
  Então o log contém: timestamp, service_name, action, truck_id
  E o log contém: duration_ms, status (success/failure)

# ── Step Definition ──
@then('todos os logs contêm "abc-123"')
def step_assert_correlation(context, caplog):
    for record in caplog.records:
        assert "abc-123" in record.message

Por que rastreabilidade importa: Em integrações distribuídas, quando algo falha, o Correlation ID permite rastrear a cadeia completa de chamadas entre serviços.
