PARTE 1

BDD & Gherkin

Behavior-Driven Development — Do requisito à especificação executável

BDD

O que é BDD?

Behavior-Driven Development

BDD é uma prática que conecta requisitos de negócio a código executável usando uma linguagem que todos entendem — negócio, QA e devs.

groups

Colaboração

PO, dev e QA escrevem cenários juntos

description

Documentação Viva

Os cenários são a documentação E o teste

TDD vs BDD

TDD (Test-Driven Development)

Foco no como implementar. Escrito por devs, para devs. Testa unidades de código.

BDD (Behavior-Driven Development)

Foco no comportamento esperado. Escrito em linguagem natural. Testa requisitos de negócio.

GHERKIN

Gherkin — A Linguagem do Negócio

Gherkin é uma linguagem estruturada para descrever comportamentos de software em linguagem natural. Usa palavras-chave: Given, When, Then

Given

"Dado que..."

Pré-condição. O estado inicial do sistema antes da ação.

Ex: "Dado uma nota fiscal válida com NCM 84713012"

When

"Quando..."

Ação. O evento que dispara o comportamento sendo testado.

Ex: "Quando o sistema envia a nota para a SEFAZ"

Then

"Então..."

Resultado esperado. O que o sistema deve produzir como resposta.

Ex: "Então recebe protocolo de autorização"

GHERKIN

Anatomia de um Arquivo .feature

Gherkin features/rf01_enviar_nota.feature 
Feature: Envio de Nota Fiscal para SEFAZ
  Como operador fiscal do sistema ASIS
  Quero enviar notas fiscais eletronicamente
  Para cumprir obrigações fiscais de forma automatizada

  Background:
    Given o sistema ASIS está autenticado na SEFAZ
    And existe um certificado digital válido

  @rf01 @sucesso
  Scenario: Envio bem-sucedido de nota fiscal
    Given uma nota fiscal válida com valor de R$ 1.500,00
    And todos os campos obrigatórios estão preenchidos
    When o sistema envia a nota para a SEFAZ
    Then deve receber protocolo de autorização
    And o status da nota deve ser "autorizada"

  @rf01 @excecao
  Scenario: Rejeição por dados incompletos
    Given uma nota fiscal sem o campo CNPJ do destinatário
    When o sistema tenta enviar a nota
    Then deve retornar erro de validação
    And a mensagem deve indicar "CNPJ obrigatório"
lightbulb

Feature = requisito. Scenario = caso de teste. Background = pré-condição comum a todos os cenários. Tags (@rf01) = rastreabilidade com o requisito.

BDD

De Gherkin para Código — pytest-bdd

Gherkin (.feature) O que o negócio lê 
Scenario: Envio bem-sucedido
  Given uma nota fiscal válida
  When o sistema envia a nota
  Then recebe protocolo
Python (pytest-bdd) O que a máquina executa 
from pytest_bdd import scenarios, given, when, then

scenarios("rf01_enviar_nota.feature")

@given("uma nota fiscal válida")
def nota_valida():
    return {"cnpj": "12345678000199",
            "valor": 1500.00}

@when("o sistema envia a nota")
def enviar(nota_valida, mock_sefaz):
    mock_sefaz.return_value = {
        "protocolo": "135240001234567"}
    return processar(nota_valida)

@then("recebe protocolo")
def verificar(resultado):
    assert resultado.protocolo

A mágica: O arquivo .feature é legível pelo PO. O arquivo .py é executável pelo CI/CD. Juntos, são o artefato de "Requisitos como Código". Instale com: pip install pytest-bdd

Tags e Organização — Rastreabilidade

Tags para Rastreabilidade

@rf01 — Requisito Funcional 01

@rnf02 — Requisito Não Funcional 02

@seguranca — Atributo de qualidade

@sucesso — Caminho feliz

@excecao — Cenário de erro

@resistir — Tática de robustez

Executando por Tag

# Rodar só testes de RF01
pytest --tags=rf01

# Rodar só testes de segurança
pytest --tags=seguranca

# Rodar cenários de exceção
pytest --tags=excecao

# Rodar só a tática RESISTIR
pytest --tags=resistir

Filtrar testes por tag permite executar subconjuntos específicos no CI/CD.

Estrutura sugerida: features/ (arquivos .feature) + tests/step_defs/ (implementações pytest-bdd). Cada feature = um requisito. Cada scenario = um caso de teste.

BDD

Por que BDD para Aferição de Qualidade?

translate

Legibilidade

O arquivo .feature é compreensível por qualquer stakeholder, não apenas devs. É a documentação do requisito.

play_circle

Executabilidade

Cada cenário Gherkin é executado automaticamente pelo pytest-bdd. Requisito documentado = requisito testado.

link

Rastreabilidade

Tags (@rf01, @rnf02) conectam cada teste ao requisito original. Sabe-se exatamente o que está coberto.

Aferição: O código BDD não implementa a funcionalidade — ele afere se a funcionalidade atende ao requisito. É a mesma ideia do artefato: inspecionar, não construir.

ISO 25010 & Táticas de Robustez

Resistir, Detectar, Recuperar — Codificando atributos de qualidade

ISO 25010

ISO/IEC 25010 — Qualidade de Produto de Software

Adequação Funcional

O sistema faz o que deveria? (RF)

Eficiência de Desempenho

Tempo de resposta, throughput, uso de recursos

Confiabilidade

Maturidade, disponibilidade, tolerância a falhas, recuperabilidade

Segurança

Confidencialidade, integridade, não repúdio, autenticidade

Compatibilidade

Coexistência e interoperabilidade

Usabilidade

Operabilidade, proteção contra erros do usuário

Manutenibilidade

Modularidade, reusabilidade, testabilidade

Portabilidade

Adaptabilidade, instalabilidade

Para o projeto ASIS: Foco em Confiabilidade, Segurança e Eficiência de Desempenho — os atributos mais críticos para um sistema fiscal.

TÁTICAS

Táticas de Robustez — Resistir, Detectar, Recuperar

Baseado no artigo "Tactics and Patterns for Software Robustness" — cada atributo de qualidade pode ser aferido através de três categorias de táticas:

shield

RESISTIR

Prevenir que o problema aconteça. Barreiras de defesa, validações, controles de acesso.

Ex: Input validation, autenticação, rate limiting, sanitização

visibility

DETECTAR

Identificar quando algo está errado. Monitoramento, alertas, health checks.

Ex: Logs estruturados, anomaly detection, heartbeat, métricas

restart_alt

RECUPERAR

Restaurar o sistema ao estado funcional. Fallback, retry, checkpoint.

Ex: Circuit breaker, retry com backoff, failover, rollback

RESISTIR

Tática RESISTIR — Prevenção como Código

Gherkin Segurança — Confidencialidade 
@rnf_seguranca @resistir
Scenario: Bloquear acesso sem token
  Given um usuário não autenticado
  When tenta acessar "/api/notas"
  Then recebe status 401 Unauthorized
  And nenhum dado é exposto

@rnf_seguranca @resistir
Scenario: Bloquear SQL injection
  Given um payload com "'; DROP TABLE--"
  When envia para "/api/notas"
  Then recebe status 422
  And o banco permanece íntegro
pytest-bdd tests/step_defs/test_resistir.py 
@given("um usuário não autenticado")
def usuario_sem_token(client):
    client.headers = {}  # Sem JWT

@when('tenta acessar "/api/notas"')
def acessar_endpoint(client):
    return client.get("/api/notas")

@then("recebe status 401 Unauthorized")
def verificar_bloqueio(response):
    assert response.status_code == 401

@then("nenhum dado é exposto")
def verificar_dados(response):
    assert "notas" not in response.json()

RESISTIR: O código afere que o sistema impede ações não autorizadas. A tática de prevenção é testada verificando que a barreira funciona.

@rnf_confiabilidade @detectar
Scenario: Health check do serviço
  Given o sistema está em operação
  When o health check é executado
  Then retorna status "healthy"
  And inclui tempo de resposta do banco
pytest-bdd tests/step_defs/test_detectar.py 
@given("o histórico de notas tem média R$ 5.000")
def setup_historico(db):
    seed_notas(db, media=5000, qtd=100)

@when("uma nota de R$ 999.999 é processada")
def processar_anomalia(client, mock_alertas):
    nota = criar_nota(valor=999999)
    return client.post("/api/notas", json=nota)

@then("um alerta de anomalia é gerado")
def verificar_alerta(mock_alertas):
    mock_alertas.assert_called_once()

@then(parsers.parse('o log registra o evento com severity "{sev}"'))
def verificar_log(caplog, sev):
    assert sev in caplog.text

DETECTAR: O código afere que o sistema identifica condições anormais. Mocks simulam o cenário de anomalia; assertions verificam que mecanismos de detecção foram ativados.

RECUPERAR

Tática RECUPERAR — Resiliência como Código

Gherkin Disponibilidade — Recuperação 
@rnf_disponibilidade @recuperar
Scenario: Failover quando SEFAZ cai
  Given a SEFAZ está indisponível
  When o sistema tenta enviar uma nota
  Then a nota é salva na fila de contingência
  And o status muda para "aguardando_reenvio"

@rnf_disponibilidade @recuperar
Scenario: Retry com backoff exponencial
  Given a SEFAZ falha nas 2 primeiras tentativas
  And volta ao normal na 3ª tentativa
  When o sistema processa a nota com retry
  Then a nota é autorizada na 3ª tentativa
  And o intervalo entre tentativas aumentou
pytest-bdd tests/step_defs/test_recuperar.py 
@given("a SEFAZ está indisponível")
def sefaz_offline(mock_sefaz):
    mock_sefaz.side_effect = ConnectionError(
        "SEFAZ fora do ar")

@when("o sistema tenta enviar uma nota")
def enviar_nota(nota_valida):
    return processar_nota(nota_valida)

@then("a nota é salva na fila de contingência")
def verificar_fila(resultado, mock_fila):
    mock_fila.assert_called_once()

@then(parsers.parse('o status muda para "{status}"'))
def verificar_status(resultado, status):
    assert resultado.status == status

RECUPERAR: O código afere que o sistema se recupera de falhas. Mock simula a falha; assertion verifica que o mecanismo de recuperação (fallback, retry, fila) foi acionado.

RNF Completo — Disponibilidade (3 Táticas)

pytest + Mock tests/test_rnf_disponibilidade.py 
class TestRNFDisponibilidade:
    """RNF: Disponibilidade — SLA 99.9% para o sistema ASIS."""

    # ── RESISTIR: Circuit Breaker ──
    @patch("app.services.sefaz_client.enviar")
    def test_resistir_circuit_breaker_abre_apos_3_falhas(self, mock_sefaz):
        """Após 3 falhas consecutivas, circuit breaker abre e evita novas chamadas."""
        mock_sefaz.side_effect = ConnectionError("timeout")
        cb = CircuitBreaker(threshold=3)

        for _ in range(3):
            cb.call(mock_sefaz)

        assert cb.state == "OPEN"
        assert mock_sefaz.call_count == 3  # Não tentou mais

    # ── DETECTAR: Health Check ──
    def test_detectar_health_check_retorna_status(self, client):
        """Endpoint /health retorna status do sistema e dependências."""
        response = client.get("/health")
        assert response.status_code == 200
        data = response.json()
        assert "database" in data
        assert "response_time_ms" in data

    # ── RECUPERAR: Retry com Backoff ──
    @patch("app.services.sefaz_client.enviar")
    def test_recuperar_retry_com_backoff(self, mock_sefaz):
        """Sistema faz retry com backoff exponencial e recupera na 3ª tentativa."""
        mock_sefaz.side_effect = [
            ConnectionError(), ConnectionError(),
            {"status": "autorizada", "protocolo": "ABC123"}
        ]
        resultado = enviar_com_retry(nota_valida, max_retries=3)
        assert resultado["status"] == "autorizada"
        assert mock_sefaz.call_count == 3

    Then requisições extras recebem 429

pytest tests/test_rnf_performance.py 
import time

def test_resistir_cache_funciona(client):
    # Primeira chamada: popula cache
    client.get("/api/notas/1")

    # Segunda chamada: deve ser mais rápida
    start = time.time()
    r = client.get("/api/notas/1")
    elapsed = (time.time() - start) * 1000

    assert r.status_code == 200
    assert elapsed < 50  # < 50ms

def test_recuperar_rate_limiting(client):
    # Enviar muitas requisições rápidas
    respostas = [client.get("/api/notas")
                 for _ in range(100)]

    status_codes = [r.status_code for r in respostas]
    assert 429 in status_codes
    # Mas as primeiras devem ter passado
    assert status_codes[0] == 200

Artigo: Tactics and Patterns for Software Robustness

auto_stories

Conceitos-Chave do Artigo

Robustez = capacidade do sistema de funcionar corretamente mesmo sob condições adversas

Táticas = decisões de design que influenciam um atributo de qualidade específico

Patterns = soluções recorrentes que implementam uma ou mais táticas

architecture

Conexão com o Artefato

1. Identificar o atributo de qualidade (ISO 25010)

2. Escolher a tática (Resistir, Detectar, Recuperar)

3. Selecionar o pattern (Retry, Circuit Breaker, etc.)

4. Escrever o cenário BDD que afere o pattern

5. Implementar com pytest-bdd + mocks

Leitura obrigatória: O artigo é a base teórica para justificar as escolhas de táticas e patterns no artefato. Cada RNF deve referenciar a tática e o pattern aplicado.

EXERCÍCIO 1

Escrever Feature BDD para um RF do Projeto

Tarefa: Escolha um RF do projeto ASIS e escreva o .feature completo com cenário de sucesso E exceção.

Template — Feature BDD para RF 
Feature: [Descrição do RF — o que o sistema deve fazer]
  Como [papel do usuário]
  Quero [ação desejada]
  Para [benefício de negócio]

  @rf[XX] @sucesso
  Scenario: [Caminho feliz — tudo funciona]
    Given  [pré-condição com dados válidos]
    When   [ação principal do RF]
    Then   [resultado esperado de sucesso]

  @rf[XX] @excecao
  Scenario: [Cenário de erro — dados inválidos ou falha]
    Given  [pré-condição com dados inválidos]
    When   [mesma ação principal]
    Then   [tratamento de erro adequado]

Dica: Pensem nos endpoints reais do ASIS. Enviar nota? Calcular imposto? Consultar status? Cada endpoint pode ser um RF com seu .feature.

EXERCÍCIO 2

Mapear as 3 Táticas para um RNF

Tarefa: Escolha um RNF do projeto ASIS e preencha o template abaixo com as 3 táticas de robustez.

Template — RNF com 3 Táticas 
RNF: [Atributo de qualidade — Ex: Segurança / Confidencialidade]
ISO 25010: [Categoria]

── RESISTIR ──
Mecanismo:  [O que impede o problema? Ex: JWT obrigatório]
Cenário:    [Given/When/Then de prevenção]
Mock:       [O que simular?]

── DETECTAR ──
Mecanismo:  [O que identifica o problema? Ex: Log de acessos]
Cenário:    [Given/When/Then de detecção]
Mock:       [O que simular?]

── RECUPERAR ──
Mecanismo:  [O que restaura? Ex: Bloqueio + notificação]
Cenário:    [Given/When/Then de recuperação]
Mock:       [O que simular?]
EXERCÍCIO 3

Implementar Teste BDD com pytest-bdd

Tarefa: Transforme o Gherkin do Exercício 1 ou 2 em código executável usando pytest-bdd + mocks.

1

Instalar dependências

pip install pytest pytest-bdd pytest-mock

2

Criar o arquivo .feature

Salvar o Gherkin do exercício anterior em features/

3

Implementar os step definitions

Cada Given/When/Then vira uma função Python com @given, @when, @then

4

Executar e verificar

pytest --tags=rf01 -v — todos os testes devem passar (verde)

Lembrem: Usem @patch para mockar a SEFAZ e outras dependências externas. O foco é aferir a lógica, não a infraestrutura.
