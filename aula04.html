PARTE 1

Verificação & Validação

Pressman, Cap. 22 — Estratégias de Teste de Software

PRESSMAN

Verificação vs. Validação

Verificação

"Estamos construindo o produto corretamente?"

Garante que o software implementa corretamente uma função específica. Foco no processo — o código faz o que a especificação diz?

Exemplos: Unit tests, code review, análise estática, linting

Validação

"Estamos construindo o produto certo?"

Garante que o software é rastreável aos requisitos do cliente. Foco no produto — o sistema entrega valor de negócio?

Exemplos: Testes de aceitação, BDD/Gherkin, validação com stakeholder

Referência: Pressman, "Engenharia de Software: Uma Abordagem Profissional", Cap. 22. Disponível na Biblioteca Virtual Integrada.

PRESSMAN

A Espiral de Testes

memory

Unit Test

Testa uma função ou método isolado. Usa mocks para dependências.

arrow_forward
hub

Integration Test

Testa a interface entre módulos. Verifica que componentes se conectam.

arrow_forward
fact_check

Validation Test

Verifica conformidade com requisitos. BDD + testes de aceitação.

arrow_forward
cloud_done

System Test

Testa o sistema integrado: performance, segurança, recuperação.

Para o artefato: RF são testados principalmente nos níveis de Unit e Validation (o sistema faz o que deveria?). RNF são testados nos níveis de Integration e System (o sistema se comporta como deveria?).

CONCEITOS

RF vs RNF — Onde Cada Um É Testado

Requisitos Funcionais (RF)

O que o sistema faz.
"Enviar nota fiscal", "Consultar status", "Calcular imposto"

Teste: Dado um input X, o output é Y?
Nível: Unit test + Validation test
Técnica: pytest + BDD/Gherkin + Mocks

Requisitos Não Funcionais (RNF)

Como o sistema se comporta.
"Responde em <200ms", "Suporta 100 req/s", "Logs rastreáveis"

Teste: Sob condição X, a qualidade Y é atendida?
Nível: Integration + System test
Técnica: JMeter, Chaos test, Security scan

ISO/IEC 25010: Functional Suitability (RF) + Performance Efficiency, Reliability, Security, Maintainability (RNF). Ambos são atributos de qualidade — ambos podem ser aferidos por código.

ESTRATÉGIA

Estrutura de um Teste Automatizado

Padrão AAA — Arrange, Act, Assert O esqueleto de todo bom teste 
def test_enviar_nota_fiscal_com_sucesso(client, mock_sefaz):
    # ARRANGE — Preparar o cenário
    nota = criar_nota_valida()
    mock_sefaz.return_value = {"status": "autorizada"}

    # ACT — Executar a ação
    response = client.post("/api/notas", json=nota)

    # ASSERT — Verificar o resultado
    assert response.status_code == 201
    assert response.json()["status"] == "autorizada"
    mock_sefaz.assert_called_once()  # Verificar que chamou a SEFAZ
lightbulb

AAA: Todo teste segue esse padrão — monte o cenário (Arrange), execute a ação (Act), verifique o resultado (Assert). Mocks entram no Arrange para simular dependências externas.

PARTE 2

Mocks — Simulando o Mundo Real

Como testar regras de negócio sem depender de serviços externos

MOCKS

Mocks: O "Dummy" dos seus Testes

O Problema

cloud_off

SEFAZ indisponível

Não dá pra testar se o serviço real está fora do ar

payments

APIs cobram por uso

Cada chamada de teste tem custo real

timer_off

Testes lentos

Suite de 2h em vez de 2s

A Solução: Mock

Lembra da paixão da Chaiany pelo Dummy no BBB 26?

O Mock é tipo o Dummy: parece uma pessoa (ou serviço), responde como uma pessoa, mas é um boneco controlado por você para testar sua lógica sem depender de ninguém "real"!

MOCKS

Anatomia de um Mock em pytest

Python / pytest-mock Exemplo: Mockando a API da SEFAZ 
from unittest.mock import MagicMock, patch

@patch("app.services.sefaz_client.enviar_nota")
def test_rf01_enviar_nota(mock_sefaz):
    """RF01: O sistema deve enviar NF para a SEFAZ."""

    # Configurar o que o mock deve retornar
    mock_sefaz.return_value = {
        "protocolo": "135240001234567",
        "status": "autorizada",
        "data_autorizacao": "2026-02-24T10:30:00"
    }

    # Executar a função que depende da SEFAZ
    resultado = processar_nota(nota_valida)

    # Verificar comportamento
    assert resultado.status == "autorizada"
    mock_sefaz.assert_called_once_with(nota_valida)
info

@patch: Substitui a função real pelo mock durante o teste. Quando o teste termina, tudo volta ao normal. O mock registra quantas vezes foi chamado e com quais argumentos.

MOCKS

Mock para Testar RNF — Disponibilidade

Python / pytest-mock RNF2: Testar mecanismo de failover 
@patch("app.services.sefaz_client.enviar_nota")
def test_rnf02_failover_quando_sefaz_cai(mock_sefaz):
    """RNF2: Quando SEFAZ falha, sistema deve entrar em contingência."""

    # Simular FALHA da SEFAZ (ConnectionError)
    mock_sefaz.side_effect = ConnectionError("SEFAZ indisponível")

    resultado = processar_nota(nota_valida)

    # Verificar que sistema entrou em contingência
    assert resultado.status == "contingência"
    assert resultado.mensagem == "Nota será enviada quando SEFAZ retornar"

@patch("app.services.sefaz_client.enviar_nota")
def test_rnf02_retry_apos_falha_temporaria(mock_sefaz):
    """RNF2: Sistema deve fazer retry após falha temporária."""

    # Primeira chamada falha, segunda sucede
    mock_sefaz.side_effect = [
        ConnectionError("timeout"),
        {"status": "autorizada", "protocolo": "123"}
    ]

    resultado = processar_nota_com_retry(nota_valida)
    assert resultado.status == "autorizada"
    assert mock_sefaz.call_count == 2  # Tentou 2 vezes
MOCKS

Tipos de Objetos Simulados

Mock

Objeto que simula comportamento E verifica como foi chamado. Ideal para testar interações.

mock.assert_called_once()

Stub

Retorna valores pré-configurados. Não verifica chamadas — só fornece dados para o teste funcionar.

stub.return_value = {...}

Fake

Implementação simplificada que funciona (ex: banco SQLite em memória em vez de PostgreSQL real).

SQLite in-memory

No artefato: Usem Mocks para simular integrações externas (SEFAZ, bancos, APIs de parceiros). Usem Fakes para o banco de dados (SQLite em memória). O objetivo é testar a lógica de negócio, não a infraestrutura.

MOCKS

Mock vs. Teste Real — Quando Usar Cada Um

Use Mock Quando...

• Dependência externa (SEFAZ, banco, email)

• Teste precisa ser rápido e determinístico

• Quer simular cenários de erro

• API tem custo por chamada

• Quer testar LÓGICA, não infraestrutura

Use Teste Real Quando...

• Quer validar a integração funciona de fato

• Teste de contrato (o formato da API mudou?)

• Teste de performance (precisa do ambiente real)

• Smoke test em ambiente de staging

• Quer testar INFRAESTRUTURA, não lógica

Seja um Dummy nos Testes: Sem emoção, sem improviso. O Mock (seu Dummy) apenas segue o script que você deu pra ele no Arrange. É a garantia de um teste determinístico e frio!

Exemplo Completo — RF como Código

pytest + Mock tests/test_rf01_cadastro_produto.py 
class TestRF01CadastroProduto:
    """RF01: O sistema deve cadastrar produto com NCM válido."""

    def test_cadastro_produto_valido(self, client):
        """Cenário: Produto com todos os campos válidos."""
        produto = {
            "codigo": "PROD-0001", "descricao": "Widget Fiscal",
            "ncm": "84713012", "preco_unitario": 99.90
        }
        response = client.post("/v2/produtos", json=produto)
        assert response.status_code == 201
        assert response.json()["codigo"] == "PROD-0001"

    def test_rejeitar_ncm_invalido(self, client):
        """Cenário de exceção: NCM com formato errado."""
        produto = {"codigo": "X", "descricao": "T",
                   "ncm": "ABC", "preco_unitario": 10}
        response = client.post("/v2/produtos", json=produto)
        assert response.status_code == 422  # NCM deve ter 8 dígitos
lightbulb

Dica: A docstring explica o requisito. O test name segue o padrão test_rf01_*. Cada teste tem cenário de sucesso E de exceção.

Exemplo Completo — RNF como Código

pytest + Mock tests/test_rnf01_auditabilidade.py 
class TestRNF01Auditabilidade:
    """RNF01: Toda transação deve gerar log rastreável."""

    def test_transacao_gera_correlation_id(self, client, seed_notas):
        """Cenário: Cada operação tem ID único para rastreabilidade."""
        r = client.get(f"/v2/notas/{seed_notas[0].id}")
        assert "x-correlation-id" in r.headers
        assert "x-response-time" in r.headers

    def test_alerta_valor_fora_sazonalidade(self, client, mock_alertas):
        """Cenário: Valor acima da média dispara alerta."""
        nota_alta = criar_nota(valor=999999.99)  # Fora do padrão
        response = client.post("/api/notas", json=nota_alta)
        mock_alertas.assert_called_once()  # Alerta disparado!
        alerta = mock_alertas.call_args[0][0]
        assert "sazonalidade" in alerta.tipo
lightbulb

Dica: O mock_alertas verifica que o mecanismo de alerta FOI chamado e com os parâmetros corretos — sem precisar de um sistema de alertas real.

ARQUITETURA

Táticas Arquiteturais — Codificando BD

replay

Retry Pattern

Retentar automaticamente operações que falharam por problemas temporários. Testável com mock que falha N vezes e depois sucede.

alt_route

Circuit Breaker

Interromper chamadas a um serviço que está falhando repetidamente. Testável com mock que monitora contagem de falhas.

backup

Fallback / Contingência

Caminho alternativo quando o principal falha. Testável com mock que simula falha → verificar que fallback é acionado.

cached

Cache Strategy

Evitar chamadas repetidas ao mesmo recurso. Testável: primeira chamada vai ao serviço (mock), segunda retorna do cache.

No artefato: Cada tática é um mecanismo arquitetural. Cada mecanismo pode ser testado com mocks simulando cenários de falha e sucesso. É isso que o enunciado pede.
