Caso Prático: Integração de Pagamentos ASIS

Contexto, requisitos e exemplos resolvidos

CASO PRÁTICO

Contexto: Integração de Pagamentos

payments

Cenário

Sistema ASIS precisa integrar com gateway de pagamento: PIX, boleto e cartão de crédito. Cada meio de pagamento tem SLA e comportamento diferente.

account_tree

Arquitetura

App ASIS → Gateway de Pagamento → BACEN (PIX) / Operadora (Cartão) / Banco (Boleto). Cada seta = ponto de integração com tática.

Requisitos implícitos: disponibilidade, latência, fallback entre meios de pagamento.

REQUISITOS

Requisitos do Caso de Pagamentos

payments

RF1: Processar pagamento via PIX

O sistema deve processar pagamento via PIX em até 5 segundos

receipt

RF2: Gerar comprovante

Gerar comprovante com número de transação único após confirmação

verified

RNF1: Disponibilidade 99.9%

O serviço de pagamentos deve manter disponibilidade de 99.9%

sync_alt

RNF2: Fallback para boleto

Quando PIX estiver indisponível, oferecer fallback automático para boleto

speed

RNF3: Latência p95 < 200ms

Consulta de status de pagamento com latência p95 inferior a 200ms

EXEMPLO RESOLVIDO

RF1 — Processar PIX

Gherkin features/pagamento_pix.feature 
@rf @pagamento @sucesso
Feature: Processar pagamento via PIX
  Como sistema de pagamentos
  Quero processar PIX em até 5s
  Para garantir experiência do usuário

  Scenario: PIX processado com sucesso
    Given um pedido de R$ 150.00
    And o gateway PIX está disponível
    When solicito pagamento via PIX
    Then o pagamento é confirmado
      em menos de 5 segundos
    And um comprovante com número
      de transação é gerado
Python steps/pagamento_pix_steps.py 
from unittest.mock import MagicMock
import time

@given('o gateway PIX está disponível')
def step_gateway_disponivel(context):
    context.gateway = MagicMock()
    context.gateway.processar_pix.return_value = {
        'status': 'aprovado',
        'transacao_id': 'TXN-001',
        'tempo_ms': 1200
    }

@when('solicito pagamento via PIX')
def step_solicitar_pix(context):
    inicio = time.time()
    context.resultado = context.gateway \
        .processar_pix(valor=context.valor)
    context.tempo = time.time() - inicio
lightbulb

Este é o nível de detalhe esperado na entrega. Mock configura resposta realista + assert verifica comportamento específico.

EXEMPLO RESOLVIDO

RNF1 — Circuit Breaker

Gherkin features/circuit_breaker.feature 
@rnf @resiliencia @falha
Feature: Circuit Breaker no Gateway
  de Pagamento
  Como sistema resiliente
  Quero proteger contra falhas do gateway
  Para manter disponibilidade 99.9%

  Scenario: CB abre após 3 falhas
    consecutivas
    Given o gateway de pagamento está
      instável
    And o circuit breaker está CLOSED
    When ocorrem 3 timeouts consecutivos
    Then o circuit breaker muda para OPEN
    And requisições são respondidas
      com fallback

  Scenario: CB tenta recuperar em
    HALF-OPEN
    Given o circuit breaker está OPEN
      há 30 segundos
    When o tempo de reset expira
    Then o circuit breaker muda para
      HALF-OPEN
    And uma requisição de teste é
      enviada ao gateway
Python steps/circuit_breaker_steps.py 
@given('o gateway de pagamento está
  instável')
def step_gateway_instavel(context):
    context.gateway = MagicMock()
    context.gateway.processar.side_effect = [
        TimeoutError("Gateway timeout"),
        TimeoutError("Gateway timeout"),
        TimeoutError("Gateway timeout"),
        {'status': 'aprovado'}
    ]

@when('ocorrem 3 timeouts consecutivos')
def step_timeouts(context):
    context.falhas = 0
    for _ in range(3):
        try:
            context.gateway.processar()
        except TimeoutError:
            context.falhas += 1
    context.cb_estado = 'OPEN'
lightbulb

RNF = tática + mecanismo + cenário de falha. O side_effect simula falhas reais do gateway.

Erros Comuns — Evite Estes

RNF sem Número

“Ser rápido” não é RNF. Use: “latência p95 < 200ms”

bug_report

Mock Genérico

MagicMock() sem side_effect não testa nada real

error

Feature sem Falha

Sem cenário @falha, você só testa o happy path

shuffle

Confundir RF com RNF

“Processar PIX” = RF. “Em até 5s” = RNF
