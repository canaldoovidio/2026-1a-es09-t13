MASSA DE TESTES

3 Categorias de Massa de Testes

sentiment_satisfied

Happy Path

Caminho feliz — tudo funciona como esperado.

test_no_deviation_detected

Distâncias abaixo de 6 km → sem alerta

warning

Edge Cases

Casos de borda — limites do comportamento.

test_analyzer_detects_only_once

6 updates com desvio → alerta disparado só 1x

error

Failure

Cenários de falha — como o sistema reage.

circuit_breaker cenários 2-3

Serviço falha → CB abre → rejeita chamadas

Cada integração do ASIS precisa das 3 categorias. Não basta testar o happy path — edge cases e failure scenarios revelam fragilidades que só aparecem em produção.

    route_optimizer=None)

EXERCÍCIO 3

Feature BDD para RNF do ASIS

Template Gherkin features/rnf_asis.feature 
# language: pt

Funcionalidade: [RNF do ASIS — ex: Resiliência na Integração SEFAZ]
  Como um sistema que depende de serviços externos
  Quero [garantir resiliência / performance / disponibilidade]
  Para [manter qualidade de serviço sob condições adversas]

  @rnf @resiliência
  Cenário: [Mecanismo ativa sob falha]
    Dado [serviço externo com comportamento normal]
    Quando [serviço começa a falhar]
    Então [mecanismo de resiliência é ativado]

  @rnf @falha
  Cenário: [Sistema degrada graciosamente]
    Dado [serviço completamente indisponível]
    Quando [requisição chega ao sistema]
    Então [resposta de fallback é retornada]

  @rnf @recuperação
  Cenário: [Sistema recupera após falha]
    Dado [serviço voltou a responder]
    Quando [nova requisição é feita]
    Então [sistema volta ao estado normal]
info

RF testa REGRA DE NEGÓCIO. RNF testa ATRIBUTO DE QUALIDADE sob condições adversas. O RF pergunta "funciona?", o RNF pergunta "funciona bem, rápido e sem quebrar?"

EXERCÍCIO 4

Steps com MagicMock

1

Criar arquivo de steps

Em features/steps/, criar rnf_asis_steps.py com @given, @when, @then

2

Mockar dependências externas

Usar MagicMock(side_effect=TimeoutError) para simular SEFAZ indisponível

3

Rodar behave

Executar e verificar que todos os cenários passam (verde)

Python features/steps/rnf_asis_steps.py (exemplo) 
from behave import given, when, then
from unittest.mock import MagicMock

@given('que o serviço SEFAZ está indisponível')
def step_sefaz_down(context):
    context.mock_sefaz = MagicMock()
    context.mock_sefaz.emitir_nfe.side_effect = TimeoutError("SEFAZ timeout")
    context.nfe_service = NFEService(sefaz_client=context.mock_sefaz)

@when('o sistema tenta emitir uma NF-e')
def step_try_emit(context):
    context.result = context.nfe_service.emitir(nota_data={"cnpj": "123"})

@then('o circuit breaker muda para o estado "open"')
def step_assert_cb_open(context):
    assert context.nfe_service.circuit_state == "open"
EXERCÍCIO 5

Teste de Carga para o ASIS

Python tests/carga/locustfile_asis.py (adaptação) 
from locust import HttpUser, task, between

class ASISUser(HttpUser):
    wait_time = between(1, 3)  # Simula intervalo entre ações do usuário

    @task(3)  # Peso 3 — rastreamento é a operação mais frequente
    def rastrear_entrega(self):
        with self.client.get("/api/entregas/CX123/rastreio",
                             catch_response=True) as response:
            if response.status_code != 200:
                response.failure(f"Status: {response.status_code}")
            else:
                response.success()

    @task(1)  # Peso 1 — emissão de NF-e é menos frequente
    def emitir_nfe(self):
        with self.client.post("/api/nfe/emitir",
                              json={"cnpj": "12345678000199"},
                              catch_response=True) as response:
            if response.status_code != 201:
                response.failure(f"Status: {response.status_code}")
            else:
                response.success()

Defina os pesos pela frequência real de uso. No ASIS, rastreamento acontece ~3x mais que emissão de NF-e. Os pesos do Locust devem refletir isso para simular carga realista.
