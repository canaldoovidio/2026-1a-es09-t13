PARTE 1

Especificação como Código

Transformando regras de negócio em código executável

O que é "Business Driver como Código"?

lightbulb

Definição

É a prática de expressar requisitos e regras de negócio na forma de código executável (testes automatizados) que pode ser rodado continuamente para aferir se o sistema atende ao que o negócio espera.

Documento Estático

✗
  • "O sistema deve responder em até 200ms"
  • Escrito uma vez, esquecido depois
  • Ninguém valida se funciona
arrow_forward

Código Executável

✓
  • assert response_time < 0.2
  • Roda a cada build (CI/CD)
  • Falha = alerta imediato

Por que Codificar Regras de Negócio?

verified

Rastreabilidade

Cada requisito tem um teste associado. Se o teste passa, o requisito é atendido.

autorenew

Aferição Contínua

O código roda em todo CI/CD. Regressões são detectadas imediatamente.

description

Documentação Viva

O teste É a documentação. Sempre atualizado, sempre verdadeiro.

groups

Comunicação entre Times

Negócio e engenharia falam a mesma língua: código que expressa regras.

format_quote

Pressman (Cap. 16): "Requisitos de qualidade devem ser especificados de forma mensurável e testável, não como declarações vagas."

O Caminho: Driver → Requisito → Código

tips_and_updates

Cada tipo de teste valida um aspecto diferente do Business Driver. O importante é que todo driver tenha pelo menos um teste associado. Vamos explorar cada um.

TIPO DE TESTE

Teste de Performance

speed

O que valida?

Tempo de resposta, throughput e estabilidade sob carga. Garante que o sistema atende os requisitos de performance mesmo com milhares de usuários simultâneos.

build

Ferramentas

JMeter (Java, mais popular), Gatling (Scala, código como teste), Locust (Python, distribuído), k6 (JavaScript moderno).

warning Caso: Black Friday 2023

Problema: E-commerce brasileiro com pico de 50k acessos/min. Sem testes de carga, o gateway de pagamento travou — R$ 2,3M em vendas perdidas em 47 minutos.

Impacto: Além da receita direta, perda de confiança do consumidor. Taxa de abandono subiu para 78%.

Correção: Testes de carga semanais com JMeter simulando 3x o pico esperado. Auto-scaling configurado com base nos resultados.

TIPO DE TESTE

Teste de Segurança

security

O que valida?

Vulnerabilidades como SQL Injection, XSS, autenticação fraca e exposição de dados. Garante que o sistema resiste a ataques conhecidos (OWASP Top 10).

build

Ferramentas

OWASP ZAP (scanner gratuito), Burp Suite (pen testing), SonarQube (SAST), Snyk (dependências vulneráveis).

warning Caso: Vazamento de Dados LGPD

Problema: Fintech expôs API sem autenticação adequada. CPFs e dados bancários de 500k clientes vazaram. Multa ANPD de R$ 50M + dano reputacional.

Impacto: Além da multa, 23% dos clientes cancelaram contas. Custo total estimado em R$ 180M.

Correção: OWASP ZAP integrado ao CI/CD. Toda PR passa por scan automático antes do merge.

TIPO DE TESTE

Teste de Regras de Negócio

rule

O que valida?

Que as regras de negócio do domínio estão implementadas corretamente. Cálculos tributários, limites de crédito, regras de pricing, compliance regulatório.

build

Ferramentas

pytest (Python), JUnit (Java), BDD/Gherkin (linguagem natural), Cucumber (multi-linguagem).

warning Caso: TaxTech e Alíquotas

Problema: Sistema tributário aplicou alíquota de ICMS errada (12% ao invés de 18% para SP) em 45k NFes. Diferença total de R$ 8,7M em crédito tributário incorreto.

Impacto: Retificação manual de cada NFe. Risco de autuação fiscal. Credibilidade da TaxTech destruída.

Correção: Cada alíquota por UF virou um @pytest.mark.business_rule executado antes de todo deploy.

TIPO DE TESTE

Teste de Resiliência

sync_problem

O que valida?

Que o sistema se recupera de falhas: queda de serviço, latência alta, perda de conexão com banco, falha de rede. A pergunta: "o que acontece quando X falha?".

build

Ferramentas

Chaos Monkey (Netflix), LitmusChaos (Kubernetes), Gremlin (plataforma), Toxiproxy (simulação de rede).

warning Caso: AWS us-east-1 (2021)

Problema: Queda parcial da região us-east-1 da AWS. Empresas sem redundância multi-região ficaram offline por 7 horas. Slack, Twitch e Disney+ afetados.

Impacto: Empresas com Chaos Engineering (como Netflix) mantiveram operação. As sem: prejuízo estimado em US$ 150M coletivo.

Correção: Testes de resiliência periódicos: desligar dependências intencionalmente e verificar que o sistema degrada graciosamente.

TIPO DE TESTE

Teste de Integração / Contrato

hub

O que valida?

Que serviços se comunicam corretamente. Em arquiteturas de microserviços, cada serviço "promete" um contrato (formato de request/response) e o teste garante que ninguém quebrou.

build

Ferramentas

Pact (CDC - Consumer Driven Contracts), Spring Cloud Contract, WireMock (mocks de API), Postman/Newman (API testing).

warning Caso: Microserviços em Produção

Problema: Time A alterou formato de resposta da API de endereços (campo "zipcode" virou "zip_code"). Time B não sabia. 50k pedidos falharam em checkout por 3 horas.

Impacto: Cada time testava isoladamente — tudo "verde". Mas a integração real falhava silenciosamente.

Correção: Contract tests com Pact: o consumidor define o que espera, o provider valida se atende. Quebra de contrato bloqueia deploy.

school "Se você não quebra de propósito, quebra por acidente."

SLA: O que Significa "99,99%"?

schedule Um mês = 720 horas

SLA Downtime/mês Downtime/ano
99% 7h 12min 3,65 dias
99,9% 43 min 8,76 horas
99,99% 4,3 min 52,6 min
99,999% 26 seg 5,26 min

Fórmula: Downtime = (1 − SLA) × 720h

cloud Deploy e SLA

Cada deploy é um risco de downtime. Se seu SLA é 99,99%, você tem apenas 4,3 minutos de tolerância por mês. Isso exige:

• Blue-Green / Canary Deploy — zero downtime

• Feature Flags — rollback instantâneo

• Health Checks — detecção automática

school

Este é um tema do Módulo M10 (DevOps). Aqui, o foco é: o SLA é um Business Driver que precisa de testes.

Composição de SLA na Cloud

Quando sua aplicação depende de múltiplos serviços, o SLA é multiplicativo, não aditivo.

dns VM (Compute) 99,95%
×
storage Banco de Dados 99,99%
×
save Disco (Storage) 99,99%
=
calculate SLA Total 99,93%

calculate O Cálculo

SLAtotal = SLA1 × SLA2 × SLA3
= 0,9995 × 0,9999 × 0,9999
= 0,9993 = 99,93%

Cada serviço adicionado reduz o SLA. Com 10 serviços de 99,99%, seu SLA cai para 99,9%.

lightbulb Reflexão para o Projeto

A ASIS TaxTech depende de: API Gateway, serviço de cálculo, banco de dados e cache Redis. Cada componente tem seu SLA.

CASO REAL

Nubank — Regras Financeiras como Código

account_balance

O Business Driver

"Zero erro em transações financeiras" para 90+ milhões de clientes. Como fintech, a confiança do cliente depende da exatidão.

science

A Solução: Contract Testing

200+ microserviços em Clojure com testes de contrato (CDC). Eliminaram E2E e adotaram testes de integração como "cavalos de batalha".

Driver → Código

Driver: Compliance financeiro 100%
Teste: Contract tests entre microserviços
Aferição: Cada serviço valida inputs/outputs via contratos
Impacto: Mataram a suite E2E que levaria "tempo infinito" para rodar

school Lição: Testes inteligentes > testes exaustivos. Qualidade da cobertura importa mais que quantidade.
CASO REAL

Nubank — Caixinha CDI e Arredondamento

savings O Problema

A "Caixinha" do Nubank remunera CDI diário. Para um depósito de R$ 10,00 a CDI de 13,25% a.a.:

rendimento_diario = 10,00 × (1,13251/252 − 1)
= 10,00 × 0,000494...
= R$ 0,00494...

O resultado tem 5+ casas decimais. Reais só têm 2 casas. O que fazer com o restante?

help_outline Arredondar ou Truncar?

ARREDONDAR
R$ 0,01
Banco paga a mais
TRUNCAR
R$ 0,00
Cliente perde

Escala: 90M clientes × R$ 0,005 de diferença/dia:

R$ 450.000/dia ≈ R$ 13,5M/mês

Arredondar a favor do cliente: banco perde R$ 13,5M/mês. Truncar: clientes perdem — e regulador pode atuar.

lightbulb

Business Driver: A regra de arredondamento precisa de teste automatizado. Uma mudança acidental no código pode custar milhões por dia no balanço.

CASO REAL

iFood — Performance sob Carga

delivery_dining

O Business Driver

"Suportar picos de milhões de pedidos". 56M de usuários ativos, 350k restaurantes, em 1.530 cidades.

warning

O Gatilho: Dia dos Namorados 2020

Durante o lockdown, a plataforma caiu durante o pico de pedidos. Após a queda, implementaram fallbacks, circuit breakers e testes de carga semanais.

Driver → Código

Driver: Disponibilidade em datas críticas
Teste: Load tests semanais com Gatling simulando comportamento real
Aferição: Testes simulam Black Friday antes de acontecer
Evolução: De testes manuais para Chaos Engineering com LitmusChaos

school Lição: Uma queda real acelerou a adoção. Melhor prevenir com testes do que remediar com desculpas.

Padrões dos Casos Reais

1

Todo Business Driver tem um teste

Não existe driver "atendido" sem evidência de código. Se não tem teste, não está garantido.

2

Testes rodam continuamente

CI/CD executa a cada commit. Qualidade não é um evento, é um processo contínuo.

3

Falhas reais aceleram a adoção

Netflix, iFood e Nubank aprenderam com incidentes. Melhor aprender com testes do que com clientes irritados.

4

Qualidade da cobertura > Quantidade

Nubank matou sua suite E2E inteira. O que importa é testar o que é crítico para o negócio.

Apache JMeter: Teste de Performance

speed

O que é?

Ferramenta open-source para teste de carga e performance. Simula múltiplos usuários simultâneos acessando a aplicação.

settings

Como funciona?

Cria Thread Groups (usuários virtuais) que executam HTTP Requests e valida as respostas com Assertions.

Exemplo: PetStore

Cenário: 100 usuários simultâneos navegando e comprando no e-commerce PetStore

Business Driver: "A loja deve suportar 100 transações por minuto com latência < 500ms"

Aferição: JMeter gera relatório com tempo de resposta, throughput e taxa de erro

link Tutorial: blog.octoperf.com/jmeter-tutorial-for-beginners

JMeter: Configuração como Código

JMeter / Test Plan (XML simplificado) petstore_load_test.jmx 
<!-- Business Driver: Suportar 100 usuários simultâneos -->
<ThreadGroup num_threads="100" ramp_up="30">
  <!-- Cenário: Navegação e compra no PetStore -->
  <HTTPSampler path="/api/v1/products" method="GET"/>
  <HTTPSampler path="/api/v1/cart/add" method="POST"/>
  <!-- Aferição: Response time < 500ms -->
  <DurationAssertion>
    <duration>500</duration>
  </DurationAssertion>
</ThreadGroup>
link

Conexão: O ThreadGroup = estímulo (100 usuários). O DurationAssertion = resposta esperada (<500ms). O JMeter executa e afere automaticamente.

JMeter: Arquitetura de Teste

Fluxo de execução de um plano de teste no JMeter

description Test Plan Configuração geral
arrow_forward
group Thread Group 100 usuários / 30s ramp-up
arrow_forward
send Samplers HTTP GET /products
arrow_forward
verified Assertions Response < 500ms
arrow_forward
assessment Listeners Relatórios e gráficos

Modos de execução

GUI Mode: para criar e debugar test plans
CLI Mode: para execução real de carga (não consome UI)

Protocolos suportados

HTTP/HTTPS, SOAP, REST, FTP, JDBC, LDAP, JMS, TCP e mais. Para o ASIS, usaremos HTTP/REST.

Componentes-Chave do JMeter

group

Thread Group

Define o número de usuários virtuais, tempo de ramp-up e quantidade de iterações. É o ponto de partida de todo teste.

send

Samplers

Requisições enviadas ao servidor: HTTP Request, JDBC Request, etc. Simulam ações reais dos usuários.

verified

Assertions

Response Assertion: status 200
Duration Assertion: < 500ms
JSON Assertion: valida payload

assessment

Listeners

Coletam e exibem resultados: Summary Report, Aggregate Graph, View Results Tree.

tune

Config Elements

Configurações reutilizáveis: HTTP Header Manager, CSV Data Set, User Variables.

schedule

Timers

Simulam tempo de espera entre requisições (think time). Tornam o teste mais realista.

pytest: Regra de Negócio como Teste

Teste 1: ICMS SP test_business_rules.py 
@pytest.mark.business_rule
def test_icms_sp(api_client):
  # REGRA: ICMS SP = 18%
  payload = {
    "uf": "SP",
    "valor": 1000.00
  }
  r = api_client.post(
    "/api/v1/calculo/icms",
    json=payload)

  assert r.status_code == 200
  assert r.json()["icms"] == 180.00
Teste 2: Rejeição test_business_rules.py 
@pytest.mark.business_rule
def test_uf_invalida(api_client):
  # REGRA: UF inválida = erro
  payload = {
    "uf": "XX",
    "valor": 1000.00
  }
  r = api_client.post(
    "/api/v1/calculo/icms",
    json=payload)

  assert r.status_code == 400
  assert "inválida" in r.text
link
